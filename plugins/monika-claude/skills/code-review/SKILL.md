# /review

對當前修改進行程式碼審查，按以下檢查清單逐項確認。

## 檢查清單

### 正確性

- 邏輯是否正確？邊界條件是否處理？
- 錯誤處理是否完整？（Go: 每個 err 都檢查；TS: try-catch 或 Result 型別）
- 是否有 race condition 或共享狀態問題？
- 輸入驗證是否完整？

### 安全性

- 是否有 SQL/NoSQL injection 風險？
- 敏感資料（密碼、token）是否暴露？
- API endpoint 是否有適當的 auth/authz 檢查？
- 是否有 XSS 風險（前端）？

### 效能

- 是否有 N+1 查詢問題？
- 大量資料是否有分頁？
- 是否有不必要的記憶體分配？
- 是否有可以快取的重複計算？

### 可讀性

- 命名是否清楚表達意圖？
- 函式是否過長？（建議 \< 50 行）
- 是否有魔術數字需要抽常數？
- 註解是否說明「為什麼」而非「做什麼」？

### 測試

- 是否有對應的測試？
- 測試是否覆蓋主要路徑和邊界情況？
- 測試命名是否清楚描述測試場景？

## 輸出格式

按嚴重程度分類回報：

- **MUST FIX**：會造成 bug、安全漏洞、資料損失
- **SHOULD FIX**：可讀性差、效能問題、缺少測試
- **NICE TO HAVE**：風格建議、小幅改善
